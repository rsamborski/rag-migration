# Product Guidelines

This document outlines the style, tone, and conventions for all content produced for the RAG Embedding Migration Framework project.

## 1. Core Principles

*   **Clarity and Precision:** Our primary audience is senior technical experts. All content must be clear, unambiguous, and technically precise. Avoid marketing fluff and vague terminology.
*   **Authority and Trust:** The content should be authoritative, well-researched, and reliable. All claims, benchmarks, and code must be tested and verifiable.
*   **Educational and Actionable:** The goal is to empower engineers. Every article should not only explain *why* but also show *how*, providing actionable steps and reusable code.

## 2. Prose Style & Tone

*   **Tone:** Professional, direct, and informative. The tone should be that of a senior engineer sharing expert knowledge with peers.
*   **Voice:** Active voice is strongly preferred over passive voice to make instructions clearer and more direct.
*   **Terminology:** Use industry-standard technical terms correctly. Define any non-standard or project-specific terms upon first use. Acronyms should be spelled out on their first appearance (e.g., "Retrieval-Augmented Generation (RAG)").

## 3. Code Conventions

*   **Clarity:** Code samples must be clean, well-commented, and easy to understand. The focus is on demonstrating a concept, not on writing the most compact code possible.
*   **Completeness:** All code required to run an example should be provided or linked to from a public repository. There should be no "magic" steps.
*   **Style:** Adhere to standard style guides for the respective languages (e.g., PEP 8 for Python, Prettier for JavaScript/TypeScript).

## 4. Visual & UX Principles (for UI examples)

*   **Function over Form:** The UI's primary purpose is to demonstrate functionality. It should be clean, simple, and utilitarian. Do not add complex styling that distracts from the core demonstration.
*   **Clarity:** Clearly label all UI elements. Provide hints or text that explain what the user is seeing and what they should do.
*   **Responsiveness:** The UI should be functional on a standard desktop screen. Full mobile responsiveness is not a primary goal for these technical demos.
